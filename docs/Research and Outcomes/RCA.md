# Root Cause Analysis: Train-Serve Skew & Architectural Pivot to LYCOS-IDS2017

## Executive Summary
During the live deployment of the Network Intrusion Detection System (NIDS), a severe train-serve skew was identified. While the LightGBM model achieved highly accurate metrics during training on the standard CIC-IDS2017 dataset, it failed to detect malicious traffic during live PCAP simulations. Extensive research and debugging revealed fundamental flaws in both the original dataset and the feature extraction toolchain (CICFlowMeter vs. PyFlowMeter). To ensure production-grade reliability, the architecture has been officially migrated to **LYCOS-IDS2017** and the **LycoSTand** extraction engine.

---

## 1. Initial Model Performance & The Inference Failure

During the model training and evaluation phase, the LightGBM classifier emerged as the best-performing model, demonstrating strong capabilities in classifying network attacks:

* **F1 Score (XGBoost):** 0.9370
* **Best Model (LightGBM):** 0.9402
* **Training Metrics:** `F1: 0.9998 | Precision: 0.9998 | Recall: 0.9998`
* **Testing Metrics:** `F1: 0.9402 | Precision: 0.9029 | Recall: 0.9990`

Despite this strong theoretical performance, the model failed completely during live simulation. When fed a `.pcap` file containing isolated, verified PortScan and DDoS attack traffic, the model flagged 100% of the packets as `BENIGN`. 

---

## 2. Root Cause Analysis (RCA)

Investigation into the false negatives revealed a major disconnect between the training data and the live inference environment. The failure was traced to two critical issues:

### A. Toolchain Mismatch (Java vs. Python)
The model was trained on CSV files generated by the original **CICFlowMeter** (written in Java). However, the live inference pipeline utilized **PyFlowMeter** (a third-party Python port). 
* **Feature Name Drift:** PyFlowMeter outputs different feature keys than the original dataset, requiring manual mapping.
* **Mathematical Skew:** Even after mapping feature names, the underlying logic used to calculate network statistics (e.g., flow timeouts, empty packet handling) differed between the Java and Python implementations. This resulted in live data scales that the model did not recognize.

### B. Fundamental Flaws in CIC-IDS2017
Further academic research revealed that the original CIC-IDS2017 dataset and the CICFlowMeter tool suffer from well-documented vulnerabilities. The dataset contains mislabeled attacks, duplicate rows, and incorrectly calculated TCP flow features. Training a model on this data fundamentally compromised the system's baseline ground truth.



---

## 3. Evaluation of Architectural Alternatives

To fix the pipeline, three distinct solutions were evaluated:

### Option 1: Re-extract & Relabel using PyFlowMeter
* **Approach:** Process the raw CIC `.pcap` files through PyFlowMeter to generate new CSVs, manually script the attack labels based on IP/Timestamps, and retrain the model.
* **Drawbacks:** Processing 10+ GB of PCAP files is highly compute-intensive. More importantly, manual labeling introduces a massive margin for human error, making the ground truth unreliable.

### Option 2: Revert to the Original CICFlowMeter (Java)
* **Approach:** Replace PyFlowMeter in the inference pipeline with the original Java-based CICFlowMeter, ensuring the live data matches the training data.
* **Drawbacks:** *Garbage In, Garbage Out.* While this solves the toolchain mismatch, it relies on a fundamentally flawed dataset. Deploying a security model trained on corrupted data defeats the purpose of a legitimate NIDS.

### Option 3: Migrate to LYCOS-IDS2017 & LycoSTand
* **Approach:** Utilize LYCOS-IDS2017, a peer-reviewed dataset built to correct the exact flaws in the CIC dataset. Replace the Python inference engine with LycoSTand, the highly optimized C/C++ tool used to create the LYCOS dataset.
* **Benefits:** Guarantees mathematically accurate labels, eliminates train-serve skew by using the exact same C-engine for both training and inference, and provides superior low-latency packet processing.

---

## 4. Final Architectural Decision

**Option 3 was selected.** The pipeline has been officially migrated to **LYCOS-IDS2017**. 

By adopting a peer-reviewed, corrected dataset and integrating the high-performance `LycoSTand` C-engine into the Python API backend, the NIDS pipeline now boasts a production-grade, mathematically sound architecture free from academic data flaws and toolchain drift.
